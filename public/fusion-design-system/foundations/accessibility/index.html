Accessibility

Accessibility

Enable users of all ability levels to understand, interact with, and navigate our products and experiences with ease.

When we design with accessibility in mind, we enable people of all abilities to interact with, understand, and navigate our products. It is also something that we are legally obligated to consider. Our products should be accessible to everyone, including those with vision, hearing, cognitive, or motor impairments.

Accessible design is everyone's responsibility, from information and user experience design to development, and on to help and support. We need to understand the users' journeys and proactively anticipate their needs.

We can do this by following the four principles of the Web Content Accessibility Guidelines

  • Perceivable: It can be perceived by at least one of a person's senses.
  • Operable: All interactions are operable through a variety of input methods.
  • Understandable: All information presented and the operation of the interface must be understandable.
  • Robust: Content can be interpreted by assistive technologies.

Structure and hierarchy

A clear and consistent hierarchy helps people who navigate the page using links or headings. Use headings and titles to outline the page so that people can see the structure and how sections relate to each other.

Be predictable and use patterns and layouts familiar to our users.

Content should be visible, so avoid putting important information inside accordions or menus.

List by the level of importance

Place items on the page in a logical order, based on their importance so people don't have to search for them.

Consistent hierarchy

Create a consistent hierarchy so that people can use alternative input methods to move through the page. Headings are in numeric order and used without skipping levels.

Learn more about managing headings

Do

Use a consistent hierarchy: H1, H2, H3, H4

Don't

Use headings out of order: H6, H2, H1, H1

Text scaling

Test your designs to ensure the layout works for different languages and assistive technologies. Consider languages like German, which require longer text fields and text wrapping. Also, consider how the UI may look at 200% magnification in the browser.

Try not to use lorem ipsum as dummy text in your designs, use text that properly represents what a user might see on the screen. A good test to perform is translating from English to German and seeing if your design still works. If not, you may want to add some information to show developers where to truncate or wrap a text element so that things stay usable.

Group similar items

Group similar items under titles or headings that clearly describe the content of the group.

Do

Group items under the title 'Worked on' clearly show documents that you have worked on.

Don't

Group items under the vague title 'Pages' show random documents.

Keyboard navigation

Not everyone uses a mouse to navigate through applications and may need to use tools such as a keyboard, mouth wand, or eye-tracking system instead. People should be able to navigate and use our products with a keyboard or screen reader. Make sure that anything 'hoverable' with a mouse is also accessible to keyboard focus and screen readers.

When working on product, ensure that a keyboard can be used to:

  • navigate
  • perform all tasks that can be completed using a mouse
  • locate where you are on the page
  • tell where the keyboard focus is on a page

Focus

Keyboard focus follows the page as the eye would scan it. Focus travels from top to bottom and left to right, moving from the most to the least important item. People can navigate applications using alternative input methods (D-pads, trackballs, keyboards, and gestures). The focus should flow in a logical order.

When assessing keyboard focus:

  • Be conscious of the order of elements on the page. Do they make sense from the user's perspective?
  • Indicate where the focus is.
  • Do not select, trigger events, or display messages on focus. People may need to navigate through all available controls before making any changes.
  • If an element that is in focus disappears, it should return to wherever it was before the user focused on that element.

Use tooltips

Tooltips can be activated via the keyboard. Just like on hover, when an element gets keyboard focus, a tooltip displays.

Validate forms inline

Validate forms inline so keyboard users don't have to navigate far to get feedback.

Ideally, design interactions to prevent errors from happening in the first place and help people fix problems as they occur.

Meaningful copy

Using consistent copy (text) makes the user interface accessible to people who use a screen reader. Screen readers help people with visual impairments by reading all visible and non-visible alt-text aloud.

All text in the UI should support accessibility, whether it is a visible label or an element like alternative text.

Our content guidelines should aid you in creating text-based content for our applications. Making sure that developers have information about alt-text is your responsibility and should be provided as part of handoff.

Be concise and use plain English

It is essential that we keep content and accessibility text as concise as possible. This can make a big difference to people who are using screen readers.

Screen readers can skip irrelevant paragraphs more easily if you keep sentences short and frontload paragraphs with important information.

Plain English helps people scan and digest information quicker. It also makes content more accessible to those who speak English as a second language.

Avoid jargon where possible and always expand acronyms when using them for the first time.

Use consistent labels

Label elements and components that have the same functionality consistently. When users see these elements in the UI, they should be able to recognize and understand the function or actions attached to them. For example, a menu item that is labeled 'All folders' should open a page with the title 'All folders'. A modal or dialog with the title 'Copy page' has a primary action button labeled 'Copy'.

Do

Modal title: 'Copy...' Button label: 'Copy'.

Don't

Modal title: 'Copy...' Button label: 'Duplicate'.

Describe what an element does

Label elements with action verbs that indicate what happens when the element is selected.

Do

Edit preferences - The text describes the outcome of the action.

Don't

Preferences - This does not clarify what will happen when the action is performed by the system.

Button labels

In buttons, describe what the action does and, if you can, reveal what will happen.

Do

Add user

Don't

Go for it!

Hyperlinks and skip links

Link text should indicate where the link will lead to. When linking text it's good to use “Learn more” along with a description of where we are taking the user.

E.g. Learn more about Accessibility

Linking the entire message gives users context about the link location, lowering their cognitive load.

Add skip links or anchors to help people navigate around pages that display a lot of content.

Images and video

For important information, use text, images should be used in a supportive way.

Provide alt-text for any images used in the UI. Do not assume developers will do this for you. This information should be shared during handoff.

Images containing information

If an image contains information that is only understandable by viewing the image, then we need to explain the content of the image using alt-text. Guidelines for writing alt-text:

  • The text should be an adequate replacement for the content of the image.
  • Try to be as succinct as possible. Aim for just a few words, a short sentence or two may be necessary in some situations.
  • Don't repeat information that is already present on the page, near the image in question.
  • Phrases like "image of ..." or "photo of ..." are not to be included. Aim to describe the meaning of the image itself.

Decorative images

If the image is used for decoration or to add visual interest, doesn't contain a link, and doesn't deliver important information, then include the alt attribute, but leave it empty.

Complex images and data visualizations

A complex image could be a diagram, table, map, or chart. Depending on how complex it is, the image may need text near it explaining the content.

Color

We comply with AA standard contrast ratios. This ensures a reasonable level of color contrast so that people with low vision can still see and use our products.

Include multiple visual cues

Do not convey information using color alone. Use multiple visual cues, stroke, weight, shape, text, or illustrations to ensure that everyone can access the information. This helps people who are fully or partially colorblind.

For example, these validation messages use color and icons together:

Do

The success message uses a tick icon and is green in color. While the error message uses a warning icon and is also red.

Don't

Color is the only visual cue to distinguish severity. The success message is green in color. The error message is red in color.

Use high contrast

High color contrast helps people who are partially or completely color blind see differences between certain colors. It helps us to create a strong visual hierarchy and improves usability for everyone. Make sure that the combination of text and background color fall below the WCAG recommended threshold ratio of 4.5:1 for standard text and 3:1 for large text (18px or above).

Note: Decorative images and disabled states are not subject to these contrast requirements.

We use a Figma plug-in called Stark to check contrast ratios. If you are using Fusion components or patterns, this has already been done for you. But, if you are doing something new, you will need to keep this in mind.

Note: When using Fusion color styles in Figma, hovering over the color before you select it will reveal a tooltip that will tell you what contrasting color you should use with it.

Do

Use established Fusion styles to ensure high-contrast color combinations that pass AA compliance.

Don't

Don't diverge from our styles while using low-contrast color combinations that fail AA compliance.

Testing and research

  • help your developer to understand why it’s important to execute your designs with accuracy
  • test the entire application and journeys using various assistive technologies and text scaling
  • include people with visual or motor impairments when testing
  • ensure the tasks in your applications can be accomplished by all users, regardless of their level of physical ability

When in doubt, refer to the Web Content Accessibility Guidelines 2.1.
