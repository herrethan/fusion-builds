Data visualization

Data visualization

Our charts and maps help us to make datasets easier to understand by highlighting patterns, trends, and outliers. Visualizations tell a story, remove noise, and highlight useful information.

Data visualization in Fusion

When working with data visualizations, we begin with a dataset and a question. We then analyze the data to find an answer to that question. Ideally, we should focus on answering a single question about the dataset.

Maintaining consistent styles and formats for our data visualizations is essential. It helps our users to understand what's being presented to them and helps us to give our users a sense that the data is being gathered and presented in a competent, accurate, and trustworthy manner.

The flow

Start with the problem

As with any design challenge, we need to start with a single, well-articulated problem that we are trying to solve. If we try to address multiple problems in one visualization, we run the risk that it will become overly complicated and difficult to understand.

Testing

Data visualizations are dynamic in nature. Testing with realistic data will reveal how effective our solution really is. Try to test with full datasets as well as smaller ones with just one or two data points. This will help to ensure that our solution can scale up and down in the future while remaining understandable.

Princliples

Accuracy

How faithful is our visualization to the dataset we are trying to present to the user, or rather, how accurate does it need to be for your use case? It is sometimes better to abstract the information so that the main point remains easy to understand.

We should aim to reduce noise by thinking about what information our users need to see and structure our visualization accordingly. This is often the case when the main aim of a given data visualization is to describe a trend or pattern. Showing every data point in such a scenario can reduce intuitiveness and increase cognitive load without adding much value.

Intuitiveness

Will our users immediately understand whats being visualized or will they need additional instructions to understand it? Aim to make data visualization as simple and intuitive as possible. If additional information may be required, make sure you include a way to access it.

Engagement

How much attention is your data visualization drawing on your page at a glance? Data visualizations often need some room and may need to be above a certain size in order to display all of the information effectively. They will also often include a range of colors.

These factors result in data visualizations drawing a lot of attention. But, depending on the flow, you may not want the user's attention to be drawn to that part of the page straight away. You may need them to focus on other information first. So, be mindful of how much attention your data visualization will draw when you look at the design of a page as a whole. Use a clear visual hierarchy on your page to help guide your users through the information being presented to them and keep them on track.

Axis and labels

Axis lines

Our axis lines should be used as guides to make data more understandable but should be unintrusive. We have dashed and solid lines, the lines can be light or dark. Aim to reduce visual noise as much as possible.

Use our light, dashed lines by default. We can use darker dashed lines to highlight a target value such as an average.

Our dark solid lines are reserved for showing where the main axes lie on the chart.

Skipping labels

Not every axis point needs to display a label. You can skip labels, resulting in a cleaner, more intuitive visualization that still gets the same information across.

Don't

: Squeeze labels together

Do

: Minimize visual noise

Label abbreviations

Text

  • Use a 12-hour format to display time (12am, 6pm)
  • Use a three-letter format for days of the week (Sun, Mon)
  • Use a three-letter format for months (Feb, Mar)
  • Use the format 'day + month' for specific days (10 Apr, 11 Apr)
  • Use the format month + year for specific months (Apr 2011, May 2017)

Numbers

  • Use standard monetary abbreviations for numbers to reduce visual noise
    • 1,250 1.25k
    • 1,000,000 1M
    • 1,000,000,000 1B

If extremely long labels are a hard requirement, we can display them diagonally but this pattern should be used as a last resort.

Formatting and localization

Please note that we do not format data based on the users current locale. The format is defined by where the data is coming from.

For example: If monitary values are coming from the US, they will be presented as american dollars and denoted using (USD) not ($).

For more information on formatting and locatization, please visit our content guidelines.

Color palettes/usage

You should be aware of our three data visualization color palettes when creating charts with Fusion:

  • Sequential (Default)
  • Categorical (Accent)
  • Divergent

Default and sequential values

Our default color palette for visualizations used our 'info' color swatch. To manage contrast, we can skip values to ensure that different elements are visually distinct.

Our normal convention is to skip every second value as shown below:

Don't

: 100, 200, 300, 400

Do

: 100, 300, 500, 700

Categorical

As with the other components in Fusion, we can use our accent palette to visualize different categories. We should never rely on color alone to convey this information but using these colors consistently should help users to understand what category an element represents without having to reference a legend.

Divergent

The divergent palettes should be used when there is a value of importance to which the data are to be compared.

The blue/orange palette has a natural association with temperature.

The green/red palette is typically used if displaying positive vs negative values.

Visualization types

Horizontal bar charts

Horizontal bar charts are advised when you need to show long category names as there is more space on the left side of the chart for labels.

Their orientation makes them easier to read since the layout echoes how we process information in the western world, from left to right and top to bottom.

Consider using a data table if you have more than 6 data points to display.

Vertical bar charts

A vertical bar chart could be a better choice if your data points are ordinal, meaning the categories have an inherent sequence from left to right.

For example:

  • Dates
  • Age ranges
  • Income brackets
  • Groups of quantities (e.g., 1-10, 11-20, etc.)
  • West to east timezones

Consider using a line chart if you have more than 30 data points to display.

Line charts

Line charts are advised for showing change over time. Use them to visualize patterns, trends, and comparisons.

Multi line charts

Line charts are suitable for comparing up to 4 discreet data sets.

Consider using a data table if you have more than 4 data sets to display.

Pie and Donut charts

Pie and donut charts can be used if the sum of the individual parts adds up to 100%, they are used to visualize how each part contributes to the greater whole.

Pie charts give a stronger sense of overall proportions whereas donut charts lead the reader's eye from arc to arc and create more of a narrative.

If you need to show additional information in the center of the chart, use a donut.

Waterfall charts

Waterfallfall charts show a running total and how we got there. The columns in waterfall charts are color-coded to show whether they represent a negative or positive change to the total.

Scatter charts

Scatter charts are used to show whether variables have a relationship or correlation with each other. If you need to see if two variables mean something when put together, a scatter chart can illustrate that relationship.

Bullet charts

Bullet charts were originally created as a replacement for traditional gauges. Bullet charts are used to track performance against a target, displaying the results as a single column or bar. They are made up of 3 parts. A target marker, a performance indicator (progress towards the target), and a background which is broken into sections to indicate what progress along the chart signifies. Hot/cold, positive or negative.

Maps

Sometimes it's better to see data in context.

Display changes as a status

We can use our tag components to show positive and negative changes as a status.

Data tables

Tables display large amounts of data. When multiple metrics, groups, and categories need to be presented together, use a data table.

Accessibility

Provide view options

Customers with vision issues may have difficulty understanding visual representations of data, even when employing assistive technologies.

Customers with motor issues may have trouble interacting with elements based on the level of their fine motor skills.

Others may just have difficulty understanding how we've presented the data because of the format we have chosen.

To cater to these types of issues, it is good practice to provide multiple views of data visualizations where possible. If you are showing users a complicated chart to visualize their data, consider also providing a data table that they can switch to or access. Either on the same page in the app or a related one.

Color usage

Color is one of our most important considerations when creating data visualizations.

Use our guidelines and components to ensure that labels, axis, and other supportive elements are always readable, even for users with low vision.

Similarly, ensure you choose colors that have sufficient contrast from each other. Remember, our charts are full of interactable elements, as such, they are subject to the our accessibility guidelines for color contrast.

Screen readers

Assistive technologies like screen readers find it difficult to parse visualizations that are created using <svg> elements. We should hide these elements from screen readers using {aria-hidden='true'} and provide a separate text equivalent to cater to users that need this additional support.
